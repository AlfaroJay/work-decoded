# Work Decoded — Naming Standard (NAMING.md)

**Locked in 2026-06-10**, immediately after the test-data purge. Every system (form → `/api/book` → Zapier → Airtable → Square → SMS) uses these exact spellings. If you add a feature, name things to match this document; if you find a mismatch, this document wins.

## 1. Session tiers

Canonical form: **Title Case** — `Standard`, `Premier`, `Crisis`, `Discovery`.

- The intake form's internal values are lowercase; **`/api/book` normalizes them to Title Case** before forwarding to Zapier (see `TIER_MAP` in `src/app/api/book/route.ts`). Nothing downstream should ever see lowercase.
- Airtable `Sessions.Session Type` has exactly these 4 options. Do not add case variants; never reintroduce `Premiere` (historical misspelling), `Initial Consultation`, or `Follow-Up` (retired Acuity-era labels).
- The `Session Price` formula is case-insensitive (`LOWER()`), so it survives accidents — but don't rely on that.
- Prices: Standard $100 · Premier $125 · Crisis $175 · Discovery $0.

## 2. Statuses

- **Sessions.Status:** `Pending` → `Accepted` → `Completed` / `Cancelled`. Title Case, these four only. `Accepted` is the automation trigger.
- **Invoices.Status:** `Draft` / `Pending` / `Sent` / `Paid`. `Pending` is what INV-1 picks up.

## 3. Reference codes (three distinct namespaces)

| Thing | Format | Example | Generated by |
|---|---|---|---|
| Booking reference (client-facing receipt of a booking) | `WD-XXXX-XXXX` | `WD-K7Q2-9XAH` | intake form, client-side |
| Package code (redeemable, pre-paid sessions) | `PKG-XXXX-XXXX` | `PKG-7K3R-92AX` | INV-2 Zap when rebuilt; manual until then |
| Invoice number | `WD-YYYY-NNN` | `WD-2026-001` | INV-0 / manual; NNN resets yearly |

- `X` = A–Z and 2–9 (no 0/O/1/I to avoid read-aloud mistakes).
- **Never** issue a package code with the `WD-` prefix — that namespace belongs to booking references. The `validate-code` allowlist (`^[A-Z0-9-]{4,32}$`) already accepts `PKG-`.
- The retired `INV-YYYY-NNN` invoice format must not be used.

## 4. Zaps

Pattern: `WorkDecoded <AREA>-<n>: <Plain-English description>` where AREA ∈ `SS` (session SMS/calendar), `INV` (invoicing), `PKG` (packages). Keep numbering stable; suffix experiments with `vN` and delete superseded versions after the new one is proven.

## 5. Airtable fields

- Field names: Title Case with spaces (`Session Date`, `Meet Link`).
- Derived/automation fields say what they are: `… Pretty` (display formatting), `… Time` (computed ISO timestamps), `… Scheduled` (Zap dedup checkboxes — set by Zaps, not humans).
- Every automation-touched field gets a description explaining who writes it and who reads it (the base already follows this — keep it up).

## 6. Dates, times, phones

- Client/consultant-facing date-times: always Eastern, always from the `Session Date Pretty` / `Session Time Pretty` formulas ("Wednesday, June 5", "4:45 PM ET"). Never hand-format dates in Zaps or code.
- Machine timestamps: ISO-8601 UTC with trailing `Z`.
- Phones: stored as 10 digits in Airtable; Twilio `To` is always `+1` + 10 digits (E.164).

## 7. Email / domain conventions

- Client-facing sender + support: `support@workdecodedhq.com` · Calendar owner: `hello@workdecodedhq.com`.
- Subdomains: `book.` (intake form), `feedback.` (consultant feedback), `l.` (short links). Marketing site at `www.`.
