# Work Decoded ‚Äî Naming Standard (NAMING.md)

**Locked in 2026-06-10**, immediately after the test-data purge. Every system (form ‚Üí `/api/book` ‚Üí Zapier ‚Üí Airtable ‚Üí Square ‚Üí SMS) uses these exact spellings. If you add a feature, name things to match this document; if you find a mismatch, this document wins.

## 1. Session tiers

Canonical form: **Title Case** ‚Äî `Standard`, `Premier`, `Crisis`, `Discovery`.

- The intake form's internal values are lowercase; **`/api/book` normalizes them to Title Case** before forwarding to Zapier (see `TIER_MAP` in `src/app/api/book/route.ts`). Nothing downstream should ever see lowercase.
- Airtable `Sessions.Session Type` has exactly these 4 options. Do not add case variants; never reintroduce `Premiere` (historical misspelling), `Initial Consultation`, or `Follow-Up` (retired Acuity-era labels).
- The `Session Price` formula is case-insensitive (`LOWER()`), so it survives accidents ‚Äî but don't rely on that.
- Prices: Standard $100 ¬∑ Premier $125 ¬∑ Crisis $175 ¬∑ Discovery $0.

## 2. Statuses

- **Sessions.Status:** `Pending` ‚Üí `Accepted` ‚Üí `Completed` / `Cancelled`. Title Case, these four only. The intake Zap sets `Pending` on every new booking; `Accepted` is the automation trigger.
- **Invoices.Status:** `Draft` / `Pending` / `Sent` / `Paid`. `Pending` is what INV-1 picks up.

## 3. Reference codes (three distinct namespaces)

| Thing | Format | Example | Generated by |
|---|---|---|---|
| Booking reference (client-facing receipt of a booking) | `WD-XXXX-XXXX` | `WD-K7Q2-9XAH` | intake form, client-side |
| Package code (redeemable, pre-paid sessions) | `PKG-XXXX-XXXX` | `PKG-7K3R-92AX` | INV-2 Zap (Code step) when a pack invoice is paid |
| Invoice number | `WD-YYYY-NNN` | `WD-2026-001` | Airtable formula on the Invoices table (`Invoice Seq` autonumber + created year) ‚Äî automatic for every row, whether INV-0 or a human created it. INV-1 passes it to Square, so Square and Airtable always show the same number. |

- `X` = A‚ÄìZ and 2‚Äì9 (no 0/O/1/I to avoid read-aloud mistakes).
- `NNN` is a continuous sequence (it does **not** reset yearly) and may have gaps where rows were deleted or invoices voided ‚Äî both are normal for the autonumber that backs it. Never type an invoice number by hand; the formula owns that field.
- **Never** issue a package code with the `WD-` prefix ‚Äî that namespace belongs to booking references. The `validate-code` allowlist (`^[A-Z0-9-]{4,32}$`) already accepts `PKG-`.
- The retired `INV-YYYY-NNN` invoice format must not be used.

## 4. Zaps

Pattern: `WorkDecoded <AREA>-<n>: <Plain-English description>` where AREA ‚àà `SS` (session SMS/calendar), `INV` (invoicing), `PKG` (packages). Keep numbering stable; suffix experiments with `vN` and delete superseded versions after the new one is proven.

## 5. Airtable fields

- Field names: Title Case with spaces (`Session Date`, `Meet Link`).
- Derived/automation fields say what they are: `‚Ä¶ Pretty` (display formatting), `‚Ä¶ Time` (computed ISO timestamps), `‚Ä¶ Scheduled` (Zap dedup checkboxes ‚Äî set by Zaps, not humans).
- Every automation-touched field gets a description explaining who writes it and who reads it (the base already follows this ‚Äî keep it up).

## 6. Dates, times, phones

- Client/consultant-facing date-times: always Eastern, always from the `Session Date Pretty` / `Session Time Pretty` formulas ("Wednesday, June 5", "4:45 PM ET"). Never hand-format dates in Zaps or code.
- Machine timestamps: ISO-8601 UTC with trailing `Z`.
- Phones: stored as 10 digits in Airtable; Twilio `To` is always `+1` + 10 digits (E.164).
